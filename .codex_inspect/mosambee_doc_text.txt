Purpose

The purpose of the document is to provide a technical reference to the development team to develop payment initiation with Mosambee App using an intent API mechanism for the android platform.

Mosambee App is compiled for a target version: Android API version 32 to support the as per the PCI security guidelines.

Security Guidelines:

With intent integration, partner app and softPOS are completely segmented and partner has no compliance implication.

However, below are a few guidelines to be followed:
Strict NDA is required.
Maintain the confidentiality of the documentation, and integration kit and restrict the circulation to any external and internal stakeholder, who isn’t intended to use it. 
Any breach of confidentiality should be reported to Mosambee immediately.
Mosambee encourages the structuring and following of the SDLC policy.
Application to be released to Mosambee integration team for QA sign-off with respect to payment integration.

Payment application will be maintained compliance by Mosambee and partner to support, as required.


High-Level Integration Summary:

364490195580





Quick Process:
Merchant application to check whether Mosambee application is installed. If app is installed, it will invoke the app otherwise it will go to google play app install.

Merchant application will call Mosambee login API from intent to generate the session key.
Merchant Application will call Mosambee payment API with parameters like amount, mobile number(Optional), description(optional) and session key(mandatory).

Mosambee application will give transaction result in response, post completing the transaction.

Password token Generation:

Merchant App has to generate the password token following the below steps:
Generate the Hash of Pin using the SHA256 algorithm as Component 1
Generate the hash of Username using SHA256 algorithm as Component 2
Apply XOR in between the Component 1 and Component 2 to generate the Password token.
Password token generated will now be sent encrypted under the AES256 key shared by Mosambee concatenated with Initial Vector used for encryption.

Algorithms to be used:
AES 256
SHA 256

Password hash Representation:
1402080153035

Note: Mosambee will guide to implement the mechanism or provide a separate AAR package which would offline generate the encrypted password tokens using the input as PIN, username, public key

Frequently Asked Questions

What is the duration of login session.

Mosambee being an intent app. It will be available in the background and will be only invoked when the payment activity is necessary and is highly advised to perform a login before any activity. However, Environment audit check API can be called with or without login activity.

How we can get the partner ID which is mentioned in the login part.

We will be sharing the partner ID, key and the common PIN

Where can I send my order id.

You can use the description parameter to send the order id.

All these permissions seem to be mandatory for integration to work, right ? 



Camera - To avoid capturing the card while tapping behind the camera

Files and media - for application logs

Phone - To see if there is an active call at the time of the transaction. This is a PCI requirement
Microphone - to avoid understanding the activity in progress. Google Android is having accessibility capable to support the special disabled. This is applicable in the case of Phone

Location - for a SoftPOS, it is required to understand the location operation, as per the PCI, and acquiring guidelines

Data is checked before performing every transaction and will not be stored. However, location collected is stored and tagged along with the transaction



How do I check whether the mobile is compliant with security requirements 

Implement healthcheck activity defined under Section 2 i.e. Environment audit check

We are getting error – System Message For signing failure

You need to login first then do a health checkup.

Is apple Pay and Google pay is supported.
